Key Finding

Quote Created fires in exactly one place — check-prices-xe.ts:731 inside the calculate() promise chain. It requires: (1) successful API quote calculation, (2) pageLeft === false, (3) checkCalculateResult() passes, and (4) quote parameters changed from last fire. Any failure in the chain — including 15+ quote error codes, network timeouts, silent errors, and page exits — blocks the event entirely.

The drop isn't just "users browsing and leaving." It's a cascade of async operations, validation checks, and error conditions across two full page transitions (Recipient List → Quote Page) with 8 points where errors are caught but not surfaced to the user.

Recommendations Ordered by estimated impact

HIGH IMPACT Add error handling to startQuoteFlow() async calls

getUserCurrency() and getActiveBalanceAccounts() at recipient-list-xe.ts:513-514 have zero error handling. Wrapping them in try/catch with fallback values would prevent silent crashes. This is likely the single biggest fixable cause of drops.

HIGH IMPACT Insert Quote Accessed between Recipients Accessed and Quote Created

Adding Quote Accessed to the funnel would split the 25% into two measurable segments: "didn't reach quote page" vs "reached quote page but calculation failed." This is the #1 diagnostic step — run the 3-step funnel query above.

HIGH IMPACT Track Quote Error events grouped by errorCode

Quote Error fires when calculate() fails. Grouping by errorCode will reveal which of the 15+ error codes are hitting recipient-first users most. The top 2-3 error codes likely explain 50%+ of the quote-page drops.

MEDIUM IMPACT Fix the loading spinner stuck on new recipient re-fetch

recipient-list-xe.ts:474-484 shows loadingProvider.show() with no catch block. Adding a try/catch with loadingProvider.hide() in the catch prevents infinite spinners.

MEDIUM IMPACT Skip RecipientDetailsPage for returning users

Unvalidated bank recipients are routed to a full details page before the quote. For returning users in the recipient-first flow, consider auto-validating or showing a lightweight inline confirmation instead of a full page with edit/delete options.

MEDIUM IMPACT Track the name-split modal as an event

The name-split validation modal (checkRecipient → recipientnameSplit.toggle()) fires with no analytics. Adding a tracking event would quantify how many users hit this unexpected interruption.
