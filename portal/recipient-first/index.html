What Is Recipient First?

The Recipient First flow flips the traditional send money journey. Instead of Quote → Recipient → Pay, users start from their saved recipient list and go Recipient → Quote → Pay.


It's the highest-converting funnel at 41.8% because users who enter this flow are returning senders with previous orders — the recipient list only shows when getLastFetchedOrders() returns results. Users with no order history skip straight to the quote page (Quote Created) and never see the recipient list.


Entry points (App Native):

  • recipientFirstHome — Home page recent recipients (tap to send)
  • sendMoneyQuoteFirst — Traditional quote-first (default/comparison)

Not yet released: Web (galileo-site MT-55665 in development) and P2P (xe-apollo MT-60039 in development)

7. New vs Returning Senders

DATA QUALITY

Filter by isFirstTransaction=true on Transaction Created. First-time senders should rarely appear in recipient-first (they have no saved recipients). If they do, it validates P2P flow or indicates data quality issues.

Expected: >95% of recipient-first transactions should be from returning users. If first-time senders appear in significant numbers, investigate whether the P2P flow is creating new recipients inline.

Flow Routing Logic (Mobile)
// home.ts:895 — User clicks recent recipient on home page
async recipientClicked(recipient: IRecipient) {
  if (AuthProvider.isLoggedIn()) {
    // Set flow name for analytics
    this.resourceProvider.setSendMoneyFlow('recipientFirstHome');

    // Determine delivery method from recipient type
    const param = {
      location: 'home-recent-recipient',
      recipient: recipient,
      sendFromRecipient: true,  // ← KEY FLAG: skips recipient selection
    };

    // P2P recipients get a separate flow
    if (recipient.values?.DELIVERY_METHOD === DeliveryMethodType.PeerToPeer) {
      this.sendFlowProvider.startPeerToPeerSendFlow(recipient);
    } else {
      this.orderProvider.openCheckPrices('recipient-creation', ...param);
    }
  } else {
    // Not logged in → login first, then retry
    this.login(() => this.recipientClicked(recipient));
  }
}

Known Issues & Gaps 1 live + 2 pre-launch
LIVE — APP

Recipients Accessed → Quote Created: 25% drop analysis

Key context: Recipients Accessed only fires for users who HAVE previous orders. Users with no recipients skip straight to Quote Created (order.ts:990-992 checks getLastFetchedOrders()). So this 25% drop is among returning users with saved recipients — a high-intent audience.

Contributing factors for the drop:

  • Browse-only users — Recipients Accessed fires on page load; some users open the list to view/edit/delete recipients, not to send money
  • Currency-filtered empty list — When a currency is set, only matching recipients show. Users may see no matching results and back out
  • Country support check — After selecting a recipient, openCheckPrices() re-checks country support. Unsupported = dead end to CountryNotSupportedPage
  • Recursive routing failures — The flow calls openCheckPrices() recursively after recipient selection. Network failures between the two calls are silent drops
  • Loading + async friction — Multiple async calls (recipients, balances, country check, currency lookup) between tap and quote page render

Since these are all returning users, the browse-only percentage is lower than a general recipient page. Real loss is likely 15-20%, making this a more significant optimization opportunity than general browsing would suggest.

LIVE — APP

Mobile Error Recovery is 4x Worse Than Web

When payment fails in the recipient-first flow, only 10-12% of mobile users retry vs 41% on web. The Change Payment Method UX on mobile needs improvement to recapture failed payments.
